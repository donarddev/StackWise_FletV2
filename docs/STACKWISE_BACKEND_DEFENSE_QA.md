# StackWise AI Backend and Code Defense Q&A Guide

This document explains the backend architecture, request flow, recommendation engine, rule-based weighted scoring logic, database layer, AI integration, important code locations, and possible backend questions for the StackWise-Flet project defense.

## 1. Project Overview

StackWise AI is a Software Stack Decision-Support System built with Python and Flet. It helps users choose a suitable programming language, framework, and SDLC model based on project requirements. It uses rule-based weighted scoring for recommendation and Ollama Llama 3.2 for AI-assisted chatbot and research support.

### What to say during defense

The simplest explanation is: StackWise AI is a project that helps users decide what technology stack to use. It is not only a chatbot. The recommendation part is rule-based and explainable, while the AI/Ollama part is used for chat and research support.

## 3. Complete Backend Request Flow

When the user clicks Generate Recommendation, this is what happens:

User fills form -> Flet page collects inputs -> Recommendation controller receives request -> Recommendation service calculates scores -> Compatibility check validates stack -> Result is saved to MySQL -> Recommendation Details page displays result

### Step-by-step flow

1. The user completes the form in [ui/pages/recommendation_page.py](../ui/pages/recommendation_page.py).
2. The page triggers `on_generate` inside [app/controllers/recommendation_controller.py](../app/controllers/recommendation_controller.py).
3. The controller converts form values into engine data using `_form_to_engine_data`.
4. The controller calls `generate_recommendation_from_request(data)`.
5. The recommendation service runs `generate_full_recommendation(data)`.
6. Inside the service, the engine normalizes the request, scores candidates, checks compatibility, and chooses the best stack.
7. The controller saves the result using `RecommendationPersistenceService`, which writes to the repository and MySQL database.
8. The controller stores the saved ID in session and redirects the user to the recommendation result route.
9. [app/controllers/recommendation_result_controller.py](../app/controllers/recommendation_result_controller.py) reads the recommendation ID and loads the saved record for display.

## 5. Recommendation Engine

The recommendation engine is rule-based and uses weighted scoring. It is not machine learning. Each candidate language, framework, and SDLC model starts with a score. Project inputs add points, reduce points, or apply penalties.

### Formula

Final Score = Base Score + Feature Match + Context Fit + Preference Bonus - Mismatch Penalty

### Simple explanation

The engine checks many candidate stacks. If a stack matches the project type, platform, features, timeline, complexity, and experience level, it gets more points. If it conflicts with the project profile, it gets fewer points or is removed from the final ranking.

### What the engine actually does

- Builds a normalized request object
- Classifies project context
- Scores languages, frameworks, and SDLC models
- Applies compatibility rules between language and framework
- Ranks valid stacks
- Picks the highest valid result
- Calculates confidence
- Produces alternatives, risks, skill gaps, and roadmap suggestions

## 6. Rule-Based Weighted Scoring

### Core idea

Rule-based scoring means the system uses predefined rules to decide which stack fits best. Weighted scoring means some factors matter more than others.

### Common Q&A

|---|---|---|---|

| What is weighted scoring? | Some factors count more than others. | Important factors such as project type and platform have more influence than minor preferences. | [app/services/recommendation_service.py](../app/services/recommendation_service.py) |
| Why use rule-based scoring instead of machine learning? | It is easier to explain and test. | The system needs explainability, and it does not require a large verified dataset. ### Beginner-friendly explanation

Think of the system like a judge scoring candidates in a competition. Each stack gets points if it matches the project. A web project, for example, will usually score web-friendly technologies higher. A mobile project will score mobile-friendly technologies higher. The winner is the stack with the best fit, not the most popular stack.
